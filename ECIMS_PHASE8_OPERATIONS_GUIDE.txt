2) FUTURE PATCH UPDATE RELEASE - "App update kahan se hogi"
------------------------------------------------------------

Your app update comes from two build artifacts:

A) Backend update source
- Code path: ecims2/server/*
- Runtime DB: SQLite file configured by ECIMS_DB_PATH
- Deploy model:
  1. Pull latest backend code.
  2. Restart backend service/process.
  3. Startup runs init_db() and additive schema creation automatically.

B) Frontend update source
- Code path: ecims_admin/src/*
- Build artifact: ecims_admin/dist/*
- Deploy model:
  1. Run frontend build.
  2. Replace served static files with latest dist.
  3. Clear browser cache or version static assets (Vite hash already helps).

Recommended release commands:
- Backend tests:
  cd ecims2/server
  python -m unittest tests.test_phase15_ops_control_plane

- Frontend build:
  cd ecims_admin
  npm run build

- Optional broader backend regressions:
  cd ecims2/server
  python -m unittest tests.test_phase11_remote_actions tests.test_phase12_schedules tests.test_phase13_enrollment tests.test_phase14_evidence_vault tests.test_phase15_ops_control_plane

3) BACKUP / STATE PRESERVATION STRATEGY
------------------------------------------------------------

Phase 8 adds in-app snapshot backups:
- API:
  - POST /api/v1/admin/ops/state-backups
  - GET  /api/v1/admin/ops/state-backups
  - GET  /api/v1/admin/ops/state-backups/{backup_id}
- UI:
  Change Control page -> "State Backups" card -> Create Backup / View / Download JSON

Backup scopes:
- CONFIG_ONLY:
  user/config/control-plane oriented tables (safe for frequent pre-update snapshots)
- FULL:
  broader operational state (agents/tasks/evidence related tables included)

Sensitive field behavior:
- include_sensitive=false (default): secret/hash columns removed
- include_sensitive=true: secret/hash columns included (highly restricted use)

IMPORTANT for production safety:
- In-app JSON snapshot is for operational rollback visibility.
- For full disaster recovery always take a raw SQLite file backup also.
  Example approach (service stop + file copy):
  1) Stop backend service
  2) Copy DB file (ECIMS_DB_PATH) to secure backup location
  3) Start service

Recommended pre-patch sequence:
1. Create CONFIG_ONLY snapshot from UI.
2. Create raw SQLite file backup.
3. Deploy patch.
4. Smoke-test login + key admin pages.


------------------------------------------------------------
4) PHASE 8 WORKFLOW NOTES
------------------------------------------------------------

Playbooks:
- Supports AUTO, MANUAL, TWO_PERSON approval modes.
- TWO_PERSON enforces two distinct approvers.
- Approver cannot be same as requester.

Change Control:
- HIGH/CRITICAL requests auto-require two approvals.
- Requester cannot self-approve.

Break-glass:
- Session creation requires current password re-auth.
- Session is time-bound and auto-expires.
- Revoke action is audit logged.
